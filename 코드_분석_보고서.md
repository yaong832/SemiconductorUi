## 1. 프로젝트 개요

### 1.1 프로젝트 목적
반도체 제조 공정을 제어하고 모니터링하는 통합 제어 시스템으로, 웨이퍼를 FOUP A에서 로드하여 각 챔버(Chamber A, B, C)에서 공정을 수행한 후 FOUP B로 언로드하는 전체 공정을 자동으로 제어합니다.

### 1.2 기술 스택
- **프레임워크**: .NET Framework (WinForms)
- **언어**: C#
- **하드웨어 통신**: IEG3268 DLL (EtherCAT)
- **데이터 저장**: XML 파일 (Users.xml, Recipes.xml, Config.xml)
- **테스트 프레임워크**: MSTest

### 1.3 프로젝트 규모
- **총 C# 파일 수**: 약 76개 (Designer.cs 제외)
- **주요 파일**: Form1.cs (약 7,820줄), 다양한 컨트롤러 및 헬퍼 클래스
- **네임스페이스 구조**: 체계적으로 분리된 폴더 구조

### 2.2 계층 구조

#### 2.2.1 Presentation Layer (표현 계층)
- **Form1.cs**: 메인 폼 (약 7,820줄)
- **Forms/**: 서브 폼들 (LoginForm, RecipeManagerForm, 등)
- **Controls/**: 커스텀 컨트롤 (EquipmentDiagramControl, FoupVisualizationControl, 등)
- **EventHandlers/**: 이벤트 핸들러 분리 (9개 파일)

#### 2.2.2 Business Logic Layer (비즈니스 로직 계층)
- **Controllers/**: 핵심 비즈니스 로직
  - `TransferController`: 웨이퍼 이송 제어
  - `ChamberController`: 챔버 상태 및 공정 제어
  - `SimulationController`: 시뮬레이션 제어
  - `TmHardwareController`: TM 하드웨어 제어
- **ViewModels/**: MVVM 패턴 적용
  - `MainFormViewModel`: UI 상태 관리
- **Helpers/**: 유틸리티 및 헬퍼 클래스
  - `AlarmManager`: 알람 관리
  - `FoupManager`: FOUP 관리
  - `Form1Initializer`, `Form1Configurator`, `Form1TmProcessor`, `Form1UiUpdater`: Form1 책임 분리

#### 2.2.3 Service Layer (서비스 계층)
- **Services/**: 서비스 구현
  - `HardwareManager`: 하드웨어 초기화 및 관리
  - `LoggerService`: 로깅 서비스
  - `ServiceContainer`: 의존성 주입 컨테이너 (서비스 로케이터 패턴)
- **Repositories/**: 데이터 접근 추상화
  - `UserRepository`: 사용자 데이터 관리
  - `RecipeRepository`: 레시피 데이터 관리
  - `ConfigRepository`: 설정 데이터 관리
- **Interfaces/**: 서비스 인터페이스 정의

#### 2.2.4 Data Access Layer (데이터 접근 계층)
- XML 파일 기반 데이터 저장
- IEG3268 DLL을 통한 하드웨어 통신

## 4. 설계 패턴 및 원칙

### 4.1 적용된 설계 패턴

#### 4.1.1 Repository 패턴 ✅
- **구현**: `UserRepository`, `RecipeRepository`, `ConfigRepository`
- **장점**: 데이터 접근 로직 캡슐화, 테스트 용이성
- **평가**: 인터페이스 기반으로 잘 구현됨

#### 4.1.2 Service Locator 패턴 ✅
- **구현**: `ServiceContainer`
- **장점**: 의존성 관리 단순화
- **단점**: 의존성 숨김 (DI 프레임워크 권장)
- **평가**: WinForms 환경에서 실용적이지만, 향후 DI 프레임워크 도입 권장

#### 4.1.3 MVVM 패턴 (부분적) ✅
- **구현**: `MainFormViewModel`
- **장점**: UI와 비즈니스 로직 분리
- **평가**: 부분적으로 적용되어 있으나, 완전한 MVVM은 아님

#### 4.1.4 State Machine 패턴 ✅
- **구현**: `TransferController.TmPhase` enum
- **장점**: 복잡한 상태 전환 관리
- **평가**: TM 이송 프로세스를 명확하게 모델링

#### 4.1.5 Strategy 패턴 (암묵적) ⚠️
- **구현**: 하드웨어 모드와 시뮬레이션 모드 분기
- **평가**: 명시적인 Strategy 패턴 적용 권장

#### 4.1.6 Observer 패턴 ✅
- **구현**: `INotifyPropertyChanged`, 이벤트 핸들러
- **평가**: 표준 .NET 패턴 잘 활용

### 4.2 SOLID 원칙 준수도

#### 4.2.1 Single Responsibility Principle (SRP) ⚠️
- **Form1.cs**: 여전히 많은 책임을 가짐 (약 7,820줄)
- **개선**: Helper 클래스로 일부 분리되었으나 추가 분리 필요
- **평가**: 부분적으로 준수

#### 4.2.2 Open/Closed Principle (OCP) ✅
- **인터페이스 기반 설계**: `ITransferService`, `IChamberService` 등
- **확장 가능성**: 새로운 구현체 추가 용이
- **평가**: 잘 준수됨

#### 4.2.3 Liskov Substitution Principle (LSP) ✅
- **인터페이스 구현**: Repository 인터페이스 구현
- **평가**: 잘 준수됨

#### 4.2.4 Interface Segregation Principle (ISP) ✅
- **세분화된 인터페이스**: 각 서비스별 인터페이스 분리
- **평가**: 잘 준수됨

#### 4.2.5 Dependency Inversion Principle (DIP) ⚠️
- **Service Locator 사용**: 의존성 주입보다 서비스 로케이터 패턴 사용
- **개선**: DI 프레임워크 도입 권장
- **평가**: 부분적으로 준수

## 5. 코드 품질 평가

### 5.1 코드 구조

#### 강점 ✅
1. **명확한 폴더 구조**: 기능별로 잘 분리됨
2. **네임스페이스 분리**: 각 계층별 네임스페이스 구분
3. **인터페이스 활용**: 추상화 잘 구현
4. **이벤트 핸들러 분리**: 9개 파일로 분리하여 가독성 향상

#### 개선 필요 ⚠️
1. **Form1.cs 크기**: 약 7,820줄로 너무 큼
2. **메서드 길이**: 일부 메서드가 너무 김
3. **중첩 클래스**: 일부 클래스 내부에 너무 많은 중첩 클래스

### 5.2 코드 가독성

#### 강점 ✅
1. **XML 문서 주석**: 대부분의 공용 메서드에 주석 존재
2. **명명 규칙**: 일관된 명명 규칙 사용
3. **리팩토링**: Helper 클래스로 책임 분리 시도

#### 개선 필요 ⚠️
1. **매직 넘버**: 일부 하드코딩된 값 존재 (AppSettings로 일부 개선됨)
2. **복잡한 조건문**: 일부 복잡한 if-else 문
3. **주석 부족**: 일부 복잡한 로직에 주석 부족

### 5.3 에러 처리

#### 강점 ✅
1. **ExceptionHandler**: 중앙화된 예외 처리
2. **Try-Catch 블록**: 주요 메서드에 예외 처리 존재
3. **로깅**: 예외 발생 시 로그 기록

#### 개선 필요 ⚠️
1. **예외 타입 구분**: 모든 예외를 일반 Exception으로 처리하는 경우 존재
2. **예외 복구**: 일부 예외 발생 시 복구 로직 부족

### 5.4 테스트 가능성

#### 강점 ✅
1. **인터페이스 기반**: Repository 및 Service 인터페이스로 테스트 용이
2. **테스트 프로젝트**: `SemiconductorUi.Tests` 프로젝트 존재
3. **의존성 분리**: 일부 의존성이 인터페이스로 추상화됨

#### 개선 필요 ⚠️
1. **테스트 커버리지**: 테스트 파일 수가 적음
2. **Mock 객체**: 테스트에서 Mock 사용 필요
3. **통합 테스트**: 단위 테스트 외 통합 테스트 필요

### 5.5 성능

#### 강점 ✅
1. **타이머 최적화**: 하드웨어 모드와 시뮬레이션 모드별 타이머 주기 설정
2. **UI 업데이트 분리**: 고속 UI 업데이트 타이머 분리
3. **폴링 최적화**: 하드웨어 폴링 주기 설정 가능
